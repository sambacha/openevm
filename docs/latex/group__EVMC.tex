\begin{DoxyItemize}
\item Executes the code (obviously).
\item Calculates the running gas cost and manages the gas counter except the refund counter.
\item Controls the call depth, including the exceptional termination of execution in case the maximum depth is reached.
\end{DoxyItemize}\hypertarget{group__EVMC_autotoc_md4}{}\doxyparagraph{Host}\label{group__EVMC_autotoc_md4}

\begin{DoxyItemize}
\item Provides access to State.
\item Creates new accounts (with code being a result of VM execution).
\item Handles refunds entirely.
\item Manages the set of precompiled contracts and handles execution of messages coming to them. 
\end{DoxyItemize}

\doxysubsubsection{\texorpdfstring{evmc\_status\_code}{evmc\_status\_code}}
{\footnotesize\ttfamily enum \mbox{\hyperlink{group__EVMC_ga4c0be97f333c050ff45321fcaa34d920}{evmc\+\_\+status\+\_\+code}}}



The execution status code. 

Successful execution is represented by \mbox{\hyperlink{group__EVMC_gga4c0be97f333c050ff45321fcaa34d920a4bc3069fec2bab2a55355a72b7db68b7}{EVMC\+\_\+\+SUCCESS}} having value 0.

Positive values represent failures defined by VM specifications with generic \mbox{\hyperlink{group__EVMC_gga4c0be97f333c050ff45321fcaa34d920aed5b2a4afa5a47af732569445920a4a9}{EVMC\+\_\+\+FAILURE}} code of value 1.

Status codes with negative values represent VM internal errors not provided by EVM specifications. These errors MUST not be passed back to the caller. They MAY be handled by the Client in predefined manner (see e.\+g. \mbox{\hyperlink{group__EVMC_gga4c0be97f333c050ff45321fcaa34d920a2f3e0d8777f8d974ead27ae2a6eb2005}{EVMC\+\_\+\+REJECTED}}), otherwise internal errors are not recoverable. The generic representant of errors is \mbox{\hyperlink{group__EVMC_gga4c0be97f333c050ff45321fcaa34d920abb1dcac202eac2ed8fd3025645a2fb28}{EVMC\+\_\+\+INTERNAL\+\_\+\+ERROR}} but an EVM implementation MAY return negative status codes that are not defined in the EVMC documentation.

\begin{DoxyNote}{Note}
In case new status codes are needed, please create an issue or pull request in the EVMC repository (\href{https://github.com/ethereum/evmc}{\texttt{ https\+://github.\+com/ethereum/evmc}}). 
\end{DoxyNote}

Example of a function creating an instance of an example EVM implementation. 

Each EVM implementation MUST provide a function returning an EVM instance. The function SHOULD be named {\ttfamily evmc\+\_\+create\+\_\+\texorpdfstring{$<$}{<}vm-\/name\texorpdfstring{$>$}{>}(void)}. If the VM name contains hyphens replaces them with underscores in the function names.

\begin{DoxyParagraph}{Binaries naming convention}
For VMs distributed as shared libraries, the name of the library SHOULD match the VM name. The convetional library filename prefixes and extensions SHOULD be ignored by the Client. For example, the shared library with the \char`\"{}beta-\/interpreter\char`\"{} implementation may be named {\ttfamily libbeta-\/interpreter.\+so}.
\end{DoxyParagraph}
\begin{DoxyReturn}{Returns}
The VM instance or NULL indicating instance creation failure.
